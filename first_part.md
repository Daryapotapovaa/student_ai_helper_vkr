# 1 Обзор предметной области
В данной главе рассматриваются основные принципы отбора существующих инструментов автодополнения кода, проводится обзор популярных коммерческих и открытых решений, а также формируется система критериев для их сравнительного анализа. На основе проведенного сравнения обосновывается необходимость разработки собственного инструмента и выбирается оптимальный метод решения поставленной задачи, учитывающий специфику образовательного процесса.
## 1.1 Принцип отбора аналогов
Для проведения анализа предметной области и выявления преимуществ и недостатков существующих систем была сформирована воспроизводимая методика поиска и отбора аналогов. В связи с быстрым развитием больших языковых моделей (LLM) и их интеграцией в среды разработки, временной диапазон поиска был ограничен 2020–2025 годами. Такое условие позволяет исключить из рассмотрения устаревшие инструменты, основанные исключительно на абстрактных синтаксических деревьях (AST) без применения алгоритмов машинного обучения.

Отбор аналогов осуществлялся по двум основным направлениям:

•	В базе данных научных публикаций Google Scholar осуществлялся поиск с целью выявления актуальных академических исследований и концептуальных подходов. Поиск проводился по следующим запросам:

o	"LSP-based code completion";

o	"explainable AI code generation";

o	"pedagogical prompting LLM programming";

o	"code completion with inline comments".


•	На платформе GitHub проводился поиск открытых (open-source) решений, которые могут быть развернуты локально. Для отсеивания неактивных или низкокачественных проектов применялся фильтр по количеству звезд (stars ≥ 1000). Запросы включали:

o	language server code completion educational;

o	LSP explainable code generation.

Отбор проводился по строгому критерию соответствия предмету исследования. В первую очередь рассматривались инструменты, способные осуществлять автодополнение кода с пояснениями непосредственно в интегрированной среде разработки (IDE) на основе локального контекста проекта. При этом системы, требующие сложных учебных метаданных (например, истории успеваемости студента) или истории коммитов с ошибками, не рассматривались ввиду сложности их интеграции в легковесный плагин. Педагогическая адаптация в искомых решениях должна реализовываться преимущественно через промпт-инжиниринг [3].

В результате применения данной методики было отобрано пять решений, наиболее полно отражающих текущее состояние рынка и open-source сообщества. Четыре из них являются коммерческими или проприетарными (GitHub Copilot, Tabnine, IntelliCode, CodeWhisperer), а одно решение представляет собой открытый академический проект (CodeT5+), имеющий публикацию на arXiv и открытый репозиторий.
### 1.1.1 GitHub Copilot
GitHub Copilot – это передовой облачный ИИ-агент, разработанный совместно GitHub и OpenAI. Он интегрируется в большинство популярных сред разработки, включая VS Code, решения от JetBrains, Visual Studio и Neovim. Инструмент предлагает автодополнение в формате подстановки строк, целых блоков кода, функций или методов, основываясь на контексте открытого кода, комментариев и текущей позиции курсора. Copilot также поддерживает режим "чат" (Copilot Chat), где пользователь может задать вопрос, получить объяснения для выделенного фрагмента кода, а также автоматически сгенерировать unit-тесты и документацию [4].

Несмотря на высокую эффективность, применительно к учебному процессу Copilot имеет ряд существенных ограничений. Во-первых, он стремится выдать готовое, максимально оптимизированное решение, что лишает студента возможности самостоятельно прийти к ответу (нарушается педагогический принцип). Во-вторых, критическим барьером для массового внедрения инструмента в учебный процесс является его доступность. Несмотря на декларируемое наличие бесплатных студенческих лицензий, процесс академической верификации значительно усложнился, а приобретение платной коммерческой подписки в ряде регионов затруднительно в связи с финансовыми и санкционными ограничениями. В-третьих, Copilot является облачным сервисом: отправка фрагментов кода на серверы корпорации регулярно вызывает в академической среде обоснованные опасения по поводу приватности данных [5].
### 1.1.2 Tabnine
Tabnine – инструмент автодополнения, отличающийся гибридным подходом: он поддерживает как облачные, так и локальные модели машинного обучения (включая CodeLlama и GPT-совместимые сети). Система поддерживает широкий спектр IDE и языков программирования. Важной особенностью Tabnine является возможность работы в полностью изолированной среде без передачи кода внешним сервисам, что критически важно для защищённых корпоративных и закрытых университетских сред.

Tabnine предоставляет не только классическое автодополнение, но и режим "агента", включающий такие возможности, как генерация кода из текстового описания, рефакторинг, создание тестов и объяснение логики кода [6]. Однако с 2025 года сервис перестал предоставлять бесплатную версию и в данный момент распространяется исключительно в формате платной подписки, что делает его массовое внедрение в учебный процесс студентов затруднительным.
### 1.1.3 IntelliCode (Microsoft)
Microsoft IntelliCode – это инструмент предиктивного ввода и "умных" подсказок, глубоко интегрированный в среду разработки Visual Studio. Система на основе алгоритмов искусственного интеллекта анализирует локальный проект (текущий контекст, типы данных, использование символов) и помогает с автодополнением конструкций.

В отличие от больших языковых моделей (LLM), инструмент не предназначен для генерации текстовых пояснений, написания документации, тестов или поддержания режима чата. IntelliCode работает через стандартные LSP-языковые серверы и делает дополнения строго на уровне синтаксиса IDE и языка [7]. С точки зрения обучения программированию, данный инструмент лишь ускоряет набор рутинного кода, но не несет в себе педагогической ценности.
### 1.1.4 CodeWhisperer (Amazon)
Amazon CodeWhisperer – облачный ИИ-инструмент для автодополнения кода, ориентированный в первую очередь на профессиональную разработку с использованием экосистемы AWS (Amazon Web Services). При этом он применим и в общих задачах программирования. CodeWhisperer способен генерировать код на основе текстовых комментариев и окружающего контекста. Наибольшую эффективность инструмент показывает в специфических AWS-сценариях [8].

Поддержка осуществляется через расширения для популярных IDE. В отличие от Copilot, интерфейс диалогового чата здесь развит слабее или вовсе отсутствует в базовых версиях. Инструмент категорически не ориентирован на выдачу объяснений и пошаговые комментарии для новичков. Его основная и единственная функция – максимизация скорости автогенерации кода. Интеграция с LSP возможна через сторонние плагины и утилиты, однако для учебных проектов его функционал избыточен в части AWS и недостаточен в части образовательной поддержки.
### 1.1.5 CodeT5+ (open-source)
CodeT5+ – это масштабируемый open-source проект от исследователей Salesforce, предоставляющий функции генерации, автодополнения, описания, перевода и рефакторинга кода [9]. В основе CodeT5+ лежит нейросетевая модель, алгоритмы которой специально оптимизированы для понимания языков программирования и генерации текста программ. Для достижения высокой точности система была предварительно обучена на большом объеме реальных программных проектов с открытым исходным кодом.

За счет открытости архитектуры и весов модели, данный проект можно настроить под нужные задачи (fine-tuning), однако для этого требуются специализированные навыки в области Machine Learning и Data Science. Проект может быть интегрирован с LSP-сервером для взаимодействия с IDE. CodeT5+ активно поддерживался сообществом до 2024 года, но на данный момент актуальных и регулярных обновлений не получает [10].
## 1.2 Критерии сравнения аналогов
Для объективной оценки рассмотренных аналогов и обоснования необходимости разработки собственного решения была сформирована система критериев. Данные критерии отражают ключевые требования, предъявляемые к ИИ-ассистентам в контексте высшего образования.
### 1.2.1 Корректность автодополнения кода
Данный критерий оценивает базовую способность инструмента генерировать семантически корректный и синтаксически точный код в режиме автодополнения. В учебном процессе предоставление студенту синтаксически неверного кода может привести к закреплению неправильных паттернов программирования и повышению когнитивной нагрузки при поиске ошибки.

Оценка корректности проводится для базовых языковых моделей, лежащих в основе инструментов, по метрике точности Pass@1 на стандартизированном бенчмарке генерации кода HumanEval. Метрика Pass@k показывает вероятность того, что среди k сгенерированных моделью вариантов кода хотя бы один пройдет все скрытые unit-тесты.

Бенчмарк HumanEval – это классический стандарт для оценки способности ИИ генерировать код на языке Python. Он состоит из 164 оригинальных программных задач различной сложности и наборов unit-тестов для каждой из них. В качестве результатов измерений для сравнения аналогов были использованы данные, полученные в авторитетном исследовании "Beyond Single LLMs: Enhanced Code Generation via Multi-Stage Performance-Guided LLM Orchestration" [11].
### 1.2.2 Поддержка генерации пояснения
Критерий оценивает способность системы не просто выдавать код-решение, но и снабжать его текстовыми комментариями и объяснениями, интегрированными непосредственно в процесс написания.

Наличие внутристрочных (inline) пояснений является ключевым свойством для педагогической адаптации. В отличие от профессиональной разработки, где лишние комментарии могут нагружать код, для студента важно понимать, почему предложен именно этот метод или как работает сгенерированный цикл. Если инструмент не способен объяснить сгенерированный фрагмент, его образовательная ценность сводится к нулю.
### 1.2.3 Настраиваемость промпта и контроль над моделью
Для исследовательского и педагогического инструмента важно иметь возможность управлять поведением модели: задавать ей роль, указывать, должна ли она объяснять шаги, использовать ли однострочные комментарии, а также вводить запреты на использование слишком сложных, не изученных ранее библиотечных функций.

Закрытые системные промпты в коммерческих продуктах в режиме автодополнения (когда пользователь не видит, какой именно запрос IDE отправляет на сервер) полностью исключают возможность внедрять образовательные стратегии. Критерий оценивает, имеет ли разработчик доступ к изменению инструкций для нейросети.

Анализ аналогов по трем критериям (корректность автодополнения, поддержка генерации пояснений и настраиваемость промпта) показал следующее: коммерческие инструменты (GitHub Copilot, Tabnine, Amazon CodeWhisperer) демонстрируют высокое качество генерации кода, но не решают поставленную исследованием задачу. Корректность автодополнения кода для Microsoft IntelliCode не определена алгоритмически, так как инструмент не является автономной генеративной языковой моделью в классическом понимании и не может быть объективно оценен метрикой Pass@1 на HumanEval. Общая особенность рассматриваемых коммерческих аналогов – ориентированность на ускорение профессиональной разработки, а не на обучение программированию.

По результатам анализа можно выделить следующие недостатки коммерческих аналогов в контексте темы исследования:

1.	Педагогическая адаптация практически отсутствует. Инструменты не встраивают пояснения в процесс автодополнения и не адаптируют стиль и сложность реализации под уровень студента.

2.	Закрытый промпт в режиме автодополнения исключает возможность внедрения образовательных стратегий или экспериментирования с поведением модели.

Единственная система, удовлетворяющая большинству требований, – CodeT5+. Она поддерживает поясняющие комментарии, работает через LSP и позволяет использовать собственный промпт. Однако её применение в учебных задачах осложнено следующими характеристиками:

1.	Модель имеет разные версии (от 60М до 6В параметров), но для получения качественных ответов требуется версия не ниже CodeT5+ 770M, что оказывает значительную нагрузку на систему при локальном запуске;

2.	Отсутствует оптимизация под современные универсальные хостинги и провайдеры LLM (такие как Ollama для локального запуска квантованных моделей или OpenRouter для облачного доступа);

3.	Архитектура не ориентирована на интеграцию в распространённые инструменты разработки студентов.

Таким образом, в ходе обзора предметной области выявлена острая потребность в легковесном, настраиваемом и открытом расширении на базе архитектуры LSP (Language Server Protocol), которое способно генерировать автодополнения с inline-пояснениями и гибко адаптироваться под нужды студентов посредством промпт-инжиниринга.
### 1.3.1 Выбор метода решения
Для устранения выявленных недостатков существующих систем и достижения цели дипломного исследования выбрано создание собственного решения. Архитектурно оно будет реализовано как open-source расширение для редактора Visual Studio Code. Решение концептуально разделяется на два модуля:

1.	Клиентская часть (VS Code Extension): отвечает за взаимодействие с пользователем, визуализацию подсказок и сбор локального контекста текущего файла.

2.	Proxy-сервер (LSP-сервер): реализует стандартный протокол Language Server Protocol для связи со средой разработки, а также содержит модуль интеграции с любой внешней или локальной LLM.

Такой подход позволяет перенести вычислительную нагрузку из IDE на отдельный процесс. Расширение будет в фоновом режиме собирать контекст для генерации кода с inline-комментариями.

Для реализации педагогической адаптации генерируемого контента будет разработан специализированный системный промпт. Он будет инструктировать языковую модель выступать в роли "помощника преподавателя", запрещая ей выдавать сложные конструкции и обязывая дополнять фрагменты кода краткими внутристрочными пояснениями. Данный метод решения позволяет объединить высокую точность современных LLM (характерную для коммерческих продуктов) с гибкостью и педагогической направленностью.
