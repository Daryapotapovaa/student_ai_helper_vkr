# 3 Программная реализация инструмента автодополнения кода

В данной главе подробно рассматривается процесс программной реализации спроектированного инструмента на базе языка TypeScript и протокола LSP, описываются механизмы параметризации сервера, отказоустойчивости, формирования педагогических промптов, а также реализация интеграции ИИ-подсказок в пользовательский интерфейс Visual Studio Code. В завершении представлены сценарии использования разработанного решения.
## 3.1 Разработка серверной части плагина
Серверная часть (Language Server) представляет собой независимый процесс Node.js, инкапсулирующий ресурсоемкую логику анализа кода и сетевого взаимодействия с искусственным интеллектом. Разработка велась с использованием официальной библиотеки vscode-languageserver.
### 3.1.1 Инициализация сервера и параметризация
Процесс запуска плагина начинается с создания объектов connection (канал связи с редактором) и documents (менеджер открытых текстовых документов). На данном этапе ля обеспечения гибкости, безопасности и соответствия требованию об обеспечении приватности происходит конфигурирование параметров, вынесенных в переменные окружения (файл .env). На этапе инициализации сервер считывает базовый сетевой адрес API в переменную API_BASE_URL, ключ доступа в API_KEY, целевую языковую модель в MODEL_NAME и режим работы в HELP_MODE.

Для взаимодействия с языковыми моделями применяется программный интерфейс библиотеки openai. Внедрение переменной API_BASE_URL позволяет использовать эту библиотеку как универсальный клиент:

•	По умолчанию сервер направляет запросы к облачному агрегатору OpenRouter;

•	При необходимости обеспечения конфиденциальности данных адрес меняется на локальный (например, порт 1234), после чего все запросы перенаправляются к локальным нейросетевым серверам (таким как LM Studio или Ollama).

Данный подход позволяет адаптировать плагин под вычислительные возможности устройств пользователей.

Важнейшим шагом инициализации является регистрация возможностей сервера (Capabilities) через обработчик connection.onInitialize. Сервер сообщает клиентскому редактору, что он поддерживает последовательную синхронизацию документов (textDocumentSync) и готов предоставлять функции быстрых исправлений кода, активируя флаг codeActionProvider.
### 3.1.2 Реализация отказоустойчивости

Ожидание ответа от внешних генеративных сетей связано с рисками возникновения сетевых ошибок, ограничений пропускной способности серверов и высоких задержек. Для минимизации влияния данных факторов в серверную часть интегрирован механизм отказоустойчивости.

При запуске сервера строковый параметр MODEL_NAME, в котором через запятую перечислены приоритетные модели, преобразуется в массив доступных ИИ-моделей MODELS_LIST. Основная логика обработки запроса заключается в цикле, который последовательно перебирает текущую модель (currentModel) из созданного массива с установленным сетевым лимитом времени, равным 15000 миллисекунд. Если основная модель отвечает успешно, выполнение цикла принудительно прерывается. Если же модель не отвечает в заданный срок или возвращает ошибку сервера, система перехватывает исключение. В этом случае ошибка логируется, сохраняется, а система автоматически инициирует запрос к следующей резервной модели в массиве.
### 3.1.3 Реализация педагогической стратегии
Ключевой особенностью серверной логики является алгоритм формирования системного запроса, хранящегося в переменной systemPrompt, который направляет языковую модель к созданию правильно структурированного ответа.

Для обхода базовых ограничений LLM, склонных в качестве ответа выдавать системные ошибки компилятора, в промпт были заложены строгие образовательные инструкции. Модели явно запрещается дублировать системные ошибки и предписывается объяснять суть проблемы, напоминая студенту фундаментальные правила языка программирования Python. 

Для обеспечения корректной программной обработки ответа модели, в переменную systemPrompt интегрировано ограничение формата. Сервер требует от LLM ответ в формате объекта JSON без использования сторонней разметки. Ожидаемая структура JSON включает логический флаг наличия ошибки (hasError), номер проблемной строки (line), педагогический комментарий (message) и, при необходимости, исправленный фрагмент кода (suggestedCode).
### 3.1.4 Алгоритм фоновой проверки кода
Для реализации событийно-ориентированной архитектуры вместо ручного вызова плагина был использован обработчик documents.onDidSave. Данный метод срабатывает автоматически каждый раз, когда обучающийся сохраняет изменения в файле.

Алгоритм проверки начинается с извлечения полного текста актуального документа. Для того чтобы языковая модель могла точно указать место возникновения ошибки, текст предварительно обрабатывается: исходный код разбивается на массив строк, и к началу каждой строки добавляется ее порядковый номер.

Далее формируется переменная systemPrompt. Сформированные данные передаются в цикл отказоустойчивости. Запрос отправляется через метод openai.chat.completions.create с ограничением времени выполнения (timeout). При успешном получении ответа строка сохраняется в переменную aiAnswerRaw. Система очищает строку от возможных артефактов Markdown-разметки и десериализует ее в JavaScript-объект. Если поле hasError имеет значение false, сервер отправляет пустой массив диагностик, завершая проверку. Если ошибка найдена, создается объект типа Diagnostic с уровнем серьезности Warning (желтое подчеркивание). В объекте указывается точный диапазон проблемной строки (range), формируется сообщение для пользователя (message), а во внутреннее поле data сохраняются параметры для последующей генерации исправлений (режим, текст и предложенный код). Сформированный объект отправляется в редактор методом connection.sendDiagnostics.
### 3.1.5 Обработка действий пользователя
Когда студент наводит курсор на подчеркнутую строку и вызывает меню быстрых действий («желтую лампочку»), редактор отправляет на сервер запрос, который перехватывается методом connection.onCodeAction.

На первом шаге сервер фильтрует массив активных проблем в переданном контексте, оставляя только те объекты Diagnostic, источником которых является разработанный плагин (проверка поля source). Если такие объекты найдены, сервер приступает к формированию массива предложений actions типа CodeAction.

Особое внимание в алгоритме уделено сохранению синтаксической структуры языка Python. Сервер извлекает из документа точный текст проблемной строки и с помощью регулярного выражения вычисляет отступ (количество начальных пробелов или символов табуляции), сохраняя его в переменную indent.

Далее логика разветвляется в зависимости от считанного из диагностики режима mode:

•	В режиме «advice» (наводящая подсказка) сервер не формирует никаких действий для модификации кода. Взаимодействие студента с ИИ-моделью осуществляется исключительно через информационное всплывающее окно и панель проблем (Problems). Данное окно автоматически генерируется редактором на основе текстового поля message объекта Diagnostic, сформированного на этапе фоновой проверки. Такой подход предотвращает засорение исходного файла избыточными текстовыми комментариями и стимулирует обучающегося к самостоятельному исправлению синтаксиса.

•	В режиме «code» (кодовая подсказка) сервер проверяет наличие сгенерированного фрагмента в поле code и формирует действие с заголовком «Применить исправление от ИИ». Для данного действия создается объект модификации TextEdit.replace, который полностью заменяет диапазон проблемной строки на строку, присланную нейросетью, предварительно добавив к ней оригинальный отступ indent. Сформированное действие добавляется в массив actions и возвращается клиенту для отображения в графическом интерфейсе.
### 3.1.6 Очистка состояния и жизненный цикл документа
Для обеспечения комфортного пользовательского опыта необходимо оперативно снимать визуальные предупреждения после того, как студент начал исправлять ошибку.

Данная задача решается с помощью метода documents.onDidChangeContent, который отслеживает любые модификации текста в буфере редактора. Как только студент начинает вводить символы или применяет исправление от ИИ, сервер мгновенно вызывает метод connection.sendDiagnostics, передавая в него пустой массив ошибок для текущего документа. Это приводит к исчезновению волнистых подчеркиваний и очистке панели проблем. Повторная интеллектуальная проверка будет инициирована только при следующем сохранении файла.

Жизненный цикл сервера замыкается вызовом методов documents.listen и connection.listen, которые переводят процесс в режим непрерывного прослушивания входящих RPC-сообщений от клиента.
## 3.2 Разработка клиентской части
Клиентская часть инструмента представляет собой расширение для среды разработки Visual Studio Code. Ее главная архитектурная задача заключается не в обработке данных, а в инициализации фонового процесса языкового сервера, настройке каналов связи и передаче управления пользовательским интерфейсом механизмам редактора. 
### 3.2.1 Инициализация и конфигурация LSP-клиента
Жизненный цикл расширения начинается с вызова функции activate, в которую среда разработки передает контекст выполнения ExtensionContext. Внутри данной функции происходит вычисление абсолютного пути к скомпилированному исполняемому файлу серверной части.

Для настройки запуска фонового процесса формируется объект serverOptions типа ServerOptions. Важным архитектурным решением на данном этапе является выбор механизма межпроцессного взаимодействия: для обмена сообщениями между процессом редактора (Extension Host) и процессом сервера используется транспорт TransportKind.ipc. Данный протокол обеспечивает высокоскоростную передачу JSON-RPC сообщений с минимальными задержками, что критически важно для передачи объемных строковых данных (кода студента и ответов нейросети). Также в объекте задаются параметры запуска для режима отладки с указанием выделенного порта.

Параллельно формируется конфигурация самого клиента в виде объекта clientOptions типа LanguageClientOptions. Основным элементом данной конфигурации является свойство documentSelector. В нем задан фильтр: расширение должно активироваться и синхронизировать документы только в том случае, если пользователь работает с файлами, имеющими схему file и идентификатор языка python. Данное ограничение является важной оптимизацией производительности: языковой сервер не расходует ресурсы и не отправляет запросы к ИИ, если студент редактирует, например, текстовые заметки (.md) или конфигурационные файлы (.json).
### 3.2.2 Управление жизненным циклом и делегирование логики
После завершения конфигурации создается экземпляр класса LanguageClient, который принимает заданные опции, и вызывается метод start(), инициирующий асинхронный запуск серверного процесса.

Благодаря реализации на событийно-ориентированной архитектуры и использованию стандартных спецификаций Language Server Protocol, клиентская часть не выполняет регистрацию кастомных команд (Commands API) и прямого манипулирования текстовым буфером редактора. Вся работа по отрисовке желтых волнистых линий (Diagnostics), генерации всплывающих информационных окон (Hovers) для режима советов и созданию кнопок быстрого исправления (Code Actions) для режима кодовых подсказок делегирована ядру Visual Studio Code. Ядро редактора автоматически интерпретирует команды, поступающие от запущенного LSP-сервера по IPC-каналу, и применяет их к интерфейсу.

Для обеспечения корректного завершения работы инструмента предусмотрена функция deactivate. При закрытии окна редактора или отключении расширения данная функция проверяет статус клиента и вызывает метод stop(), безопасно завершая фоновый процесс сервера и освобождая выделенную оперативную память.
## 3.3 Сценарии использования
Для подтверждения соответствия разработанного инструмента заявленным функциональным и нефункциональным требованиям, а также для полной оценки инструмента были разработаны три ключевых сценария, демонстрирующих различные аспекты работы системы: педагогическую адаптивность, работу с кодом и обеспечение приватности.
### 3.3.1 Сценарий 1: Режим наводящей подсказки
Первый сценарий направлен на демонстрацию работы инструмента на тех этапах обучения, когда критически важно не давать студенту готовых ответов, а стимулировать его аналитическое мышление для самостоятельного устранения ошибок. Для этого в конфигурационном файле плагина устанавливается режим наводящей подсказки (advice).

Ход тестирования: cтудент пишет фрагмент кода на языке Python, допуская синтаксическую ошибку (например, отсутствует двоеточие после объявления цикла for или функции). Студент сохраняет файл, инициируя фоновую проверку.

### 3.3.3 Сценарий 3: Работа в изолированной среде
Третий сценарий направлен на валидацию нефункционального требования по обеспечению приватности данных, что является главным аргументом при внедрении решения в закрытую среду.

Ход тестирования: симулируется ситуация, при которой существует запрет на отправку исходных кодов на внешние облачные серверы (отключение от OpenRouter). В конфигурации плагина базовый адрес API переопределяется на локальный адрес (http://localhost:1234/v1), на котором запущена языковая модель посредством среды LM Studio.

Результат работы:

1.	Плагин успешно подключается к локальному серверу, используя стандартный формат запросов OpenAI API. Фоновая проверка и генерация подсказок выполняются корректно, при этом сетевой трафик не покидает пределы компьютера пользователя.

2.	Для проверки механизма отказоустойчивости локальный сервер ИИ принудительно отключается. При попытке студента сохранить код плагин не вызывает критического сбоя редактора. Вместо этого механизм перехвата ошибок обрабатывает сбой подключения и выводит в панель проблем информационное сообщение: "Ошибка: Локальный сервер ИИ не запущен".

Успешное прохождение данного сценария доказывает архитектурную гибкость плагина и его готовность к безопасному внедрению в реальный учебный процесс без риска компрометации данных.

# **ПРИЛОЖЕНИЕ А**
## Подготовка окружения и установка зависимостей
Поскольку архитектура проекта основана на концепции независимых модулей клиента и сервера, первичная настройка требует инициализации пакетного менеджера для обеих частей. Для работы исходного кода на компьютере должен быть установлен интерпретатор Node.js.

Процесс подготовки начинается с загрузки исходного кода из системы контроля версий. Далее, находясь в корневой директории проекта, необходимо выполнить команду npm install для установки зависимостей клиентского расширения (включая библиотеки vscode-languageclient). Аналогичную команду необходимо выполнить в директории server для загрузки серверных пакетов, таких как openai и vscode-languageserver.
## Конфигурация параметров среды
Для обеспечения работы нейросетевых функций перед компиляцией проекта или его локальным запуском необходимо создать конфигурационный файл окружения .env в директории сервера. Данный файл должен содержать следующие обязательные переменные:

•	API_BASE_URL – базовый сетевой адрес для обращения к языковым моделям (например, адрес облачного сервиса OpenRouter или http://localhost:1234/v1 для работы с локальной изолированной моделью);

•	API_KEY – ключ авторизации для доступа к API;

•	MODEL_NAME – строка с перечислением доступных моделей, используемая алгоритмом отказоустойчивости;

•	HELP_MODE – режим работы педагогического ассистента (advice для получения наводящих подсказок или code для предложений по рефакторингу).

Вынесение данных настроек в отдельный файл гарантирует безопасность учетных данных и позволяет гибко управлять поведением инструмента без необходимости внесения изменений в исходный TypeScript-код.
## Компиляция и сборка установочного пакета
Поскольку среда Visual Studio Code работает с исполняемым JavaScript-кодом, исходные файлы проекта на TypeScript должны быть предварительно скомпилированы. Для этого в корневой директории запускается команда npm run compile (или npm run watch для режима активной отладки), который транслирует файлы сервера и клиента в оптимизированный JavaScript-код, помещая результаты в директории out.

Для распространения готового расширения среди студентов используется официальная утилита командной строки vsce (Visual Studio Code Extensions). После глобальной установки данной утилиты в корневой папке проекта выполняется команда vsce package. Утилита анализирует манифест проекта (package.json), упаковывает скомпилированные файлы, исключая избыточные, и генерирует единый установочный архив с расширением .vsix.
## Установка расширения в среду разработки
Полученный файл формата .vsix является конечным продуктом, готовым к дистрибуции. Процесс установки расширения упрощен и не требует работы с терминалом.

Пользователю необходимо открыть редактор Visual Studio Code, перейти во вкладку управления расширениями (Extensions), открыть меню дополнительных действий и выбрать пункт «Установить из VSIX...» (Install from VSIX...). После выбора сгенерированного архива редактор автоматически распакует клиентскую и серверную части, инициализирует фоновые процессы и активирует функционал ИИ-помощника при открытии любого файла с исходным кодом на языке Python.
