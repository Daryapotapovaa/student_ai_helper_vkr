# 2 Проектирование архитектуры инструмента автодополнения кода
В данной главе осуществляется переход от теоретического анализа предметной области к этапу системного проектирования. На основе выявленных недостатков существующих решений формируется перечень функциональных и нефункциональных требований к разрабатываемому плагину, обосновывается выбор технологического стека, описывается общая архитектура клиент-серверного взаимодействия на базе протокола LSP, а также проектируются механизмы отказоустойчивости и педагогической адаптации генерируемого контента.

##	2.1 Формирование требований к программному решению
Проектирование программного продукта начинается с фиксации требований, которые гарантируют, что итоговое решение будет выполнять поставленные задачи. Специфика целевой аудитории (студенты, изучающие программирование) накладывает особые ограничения на функциональность и архитектуру разрабатываемого инструмента.

###	2.1.1 Функциональные требования
Функциональные требования описывают поведение системы и те задачи, которые она должна выполнять в процессе взаимодействия со студентом. Разрабатываемый инструмент должен удовлетворять следующим критериям:

1.	Сбор и передача локального контекста: система должна автоматически считывать содержимое открытого файла, определять текущую позицию курсора и передавать эти данные для формирования запроса к языковой модели.

2.	Многорежимная генерация ответов: для реализации адаптивной образовательной стратегии плагин должен поддерживать два режима работы:
    * Режим наводящей подсказки (Advice) – генерация краткого текстового совета без предоставления готового программного кода;
    * Режим автодополнения (Code) – генерация исправленного или дополненного фрагмента кода с кратким поясняющим комментарием.

3.	Безопасная интеграция в исходный код: система должна анализировать синтаксическую структуру текущей строки (в частности, отступы и табуляцию) и вставлять ответ модели на новую строку без нарушения синтаксиса и логических блоков программы.

4.	Отказоустойчивость маршрутизации: система должна уметь обрабатывать ошибки таймаута и недоступности нейросетей, автоматически перенаправляя запрос к резервным языковым моделям из заранее заданного списка.
	
###  2.1.1 Нефункциональные требования
Нефункциональные требования определяют атрибуты качества системы, такие как производительность, безопасность и масштабируемость:
1.	Обеспечение приватности данных: архитектура плагина должна предусматривать возможность отключения от облачных сервисов (таких как OpenAI API или OpenRouter) и переключения на локальные серверы (например, Ollama или LM Studio). 
2.	Асинхронность и производительность: процесс взаимодействия с языковой моделью должен выполняться асинхронно в фоновом режиме, чтобы ожидание ответа не приводило к блокировке пользовательского интерфейса среды разработки.
3.	Слабая связность компонентов: интерфейс пользователя (редактор кода) и серверная логика (формирование промптов и запросы к LLM) должны быть строго разделены.
4.	Конфигурируемость: основные параметры системы, такие как адреса серверов, ключи доступа, целевые модели и режимы работы, должны выноситься в отдельные конфигурационные файлы (переменные окружения) для удобства настройки без необходимости внесения изменений в исходный код плагина.

##	2.2 Выбор стека технологий
Для реализации описанных требований необходимо подобрать оптимальный набор инструментов и технологий. Выбор стека обусловлен стандартами современной веб-разработки, требованиями к производительности и необходимостью интеграции с редактором Visual Studio Code.
	
###  2.2.1 Язык программирования
Разработка как клиентской, так и серверной части реализована на языке TypeScript. В отличие от классического JavaScript, TypeScript обеспечивает строгую статическую типизацию, что позволяет фиксировать большинство архитектурных и синтаксических ошибок еще на этапе компиляции. Кроме того, официальные библиотеки Microsoft для создания расширений VS Code написаны на TypeScript, что обеспечивает максимальную совместимость.

### 2.2.2 Среда выполнения
В качестве среды выполнения для серверной части плагина выбран Node.js. Его механизм управления асинхронными операциями (Event Loop) подходит для реализации ресурсоемких сетевых запросов (обращений к API языковых моделей), обеспечивая высокую скорость работы ввода-вывода (I/O) без задержек интерфейса.

###	2.2.3 Протокол взаимодействия
Для обеспечения слабой связности между редактором и логикой ИИ-ассистента применяется открытый протокол LSP, разработанный компанией Microsoft. Данный стандарт позволяет вынести ресурсоемкие задачи по анализу кода и генерации подсказок в отдельный процесс (Language Server). В перспективе это позволит использовать написанный сервер не только в VS Code, но и в других совместимых IDE (Neovim, Eclipse, IntelliJ) с минимальными изменениями.

###	2.2.4 Платформа интеграции
В качестве целевой среды разработки для реализации клиентской части выбран редактор Visual Studio Code. Такой выбор обусловлен его гибкой модульной архитектурой и поддержкой протокола LSP, что соответствует спроектированной клиент-серверной модели инструмента. Предоставляемый платформой VS Code Extension API дает программный доступ к работе с текстовым буфером, управлению событиями курсора и добавлению кастомных команд. Дополнительными факторами выбора является кроссплатформенность редактора, его высокое положение на рынке [12] и широкое распространение в образовательных учреждениях, что обеспечивает низкий порог входа для обучающихся.

###  2.2.5 Сетевое взаимодействие с ИИ
Для отправки запросов к языковым моделям используется официальная библиотека openai. Выбор этого SDK обоснован тем, что структура запросов OpenAI (роли system, user, assistant, параметры messages и model) фактически стала стандартом. Использование данного формата позволяет плагину быть универсальным: с помощью изменения базового URL можно отправлять идентичные запросы как в облачные агрегаторы (OpenRouter), так и в локальные среды запуска LLM, обеспечивая заявленные требования к приватности.

##  2.3 Общая архитектура клиент-серверного взаимодействия
Архитектура разрабатываемого инструмента базируется на строгом разделении ответственности между пользовательской частью (интерфейс) и вычислительной логикой. Это достигается за счет реализации стандарта Language Server Protocol (LSP). Система состоит из двух независимых модулей, обменивающихся данными посредством межпроцессного взаимодействия клиентского расширения среды разработки и автономного языкового сервера.

###  2.3.1 Роль и задачи клиентского расширения
Клиентская часть (VS Code Extension) отвечает исключительно за интеграцию с пользовательским интерфейсом среды разработки и управление состоянием текстового редактора. В ее основные задачи входит:
1.	Регистрация команд и прослушивание событий: клиент перехватывает действия пользователя (например, вызов команды автодополнения через сочетание клавиш) и формирует запрос, содержащий идентификатор текущего документа (URI).
2.	Считывание синтаксического контекста: при получении ответа от сервера клиентская часть анализирует текущую позицию курсора и считывает строку, на которой находится разработчик. Особое внимание уделяется вычислению отступов (пробелов или табуляций) в начале строки, что критически важно для языков со строгой типизацией отступов, таких как Python. При правильном вычислении позиции вставки кода или комментария ответ нейросети не нарушит область видимости локальных блоков (циклов, условий, функций).
3.	Безопасная модификация документа: клиент проводит редактирование, осуществляя вставку полученного текста в конец текущей строки с переносом, тем самым предотвращая разрыв существующего кода.
	
###  2.3.2 Роль и задачи LSP-сервера
Серверная часть функционирует в отдельном процессе Node.js. Инкапсуляция логики на стороне сервера предотвращает зависание графического интерфейса IDE во время длительных сетевых запросов. Задачи сервера включают:
1.	Сбор полного контекста документа: сервер поддерживает синхронизацию с редактором и в момент поступления запроса извлекает полный актуальный текст модифицируемого файла.
2.	Управление конфигурацией: сервер считывает глобальные настройки плагина из переменных окружения (включая адреса, ключи доступа и список доступных языковых моделей).
3.	Выполнение запросов к LLM: сервер формирует системные промпты, оборачивает пользовательский код в необходимый формат и осуществляет асинхронные вызовы к API языковой модели. После получения ответа сервер возвращает его клиенту.

##	2.4 Проектирование механизма взаимодействия с языковыми моделями
Интеграция с большими языковыми моделями (LLM) сопряжена с такими рисками, как нестабильность сетевых соединений, ограничение количества запросов или объема данных (Rate Limits) и нарушение конфиденциальности. Для минимизации этих рисков архитектура взаимодействия с ИИ-моделями была спроектирована с учетом требований универсальности и отказоустойчивости.

###	2.4.1 Проектирование механизма взаимодействия с языковыми моделями
Существенным недостатком коммерческих ИИ-ассистентов является маршрутизация исходного кода на закрытые серверы корпораций. В разрабатываемом решении эта проблема устранена за счет параметризации базового адреса.

Взаимодействие реализовано на стандарте OpenAI API. Этот формат поддерживается большинством современных генеративных сетей. Параметризация базового адреса позволяет системе работать в двух режимах:
* 	Облачный режим (по умолчанию): запросы направляются к облачным агрегаторам (например, OpenRouter), что не требует от студента наличия мощного аппаратного обеспечения.
* 	Локальный изолированный режим: путем переопределения конфигурации на localhost запросы перенаправляются на локальные серверные решения (Ollama, LM Studio). В этом режиме код не отправляется на сторонние сервера, что полностью решает проблему приватности данных и позволяет использовать инструмент при корпоративных ограничениях.

###  2.4.2 Механизм отказоустойчивости
Ожидание ответа от нейросети является главной возможной проблемой системы. Модель может быть перегружена или временно недоступна. Для обеспечения стабильной работы учебного инструмента спроектирован механизм резервного переключения (Fallback).

Архитектура механизма основывается на использовании массива приоритетных моделей, конфигурация которых передается через переменные окружения. Алгоритм работы реализован следующим образом:
1.	Сервер отправляет запрос к основной модели из списка, устанавливая лимит ожидания (Timeout) на уровне сетевого соединения.
2.	В случае истечения времени ожидания или получения кода ошибки от сервера (например, 503 Service Unavailable), система перехватывает исключение.
3.	Процесс автоматически переключается на следующую (резервную) модель в массиве без прерывания сеанса пользователя.
4.	В случае нарушения соединения со всеми доступными моделями система возвращает пользователю сообщение об ошибке подключения в виде комментария к коду.

##  2.5 Педагогическая стратегия и адаптивный промпт-инжиниринг
Основным отличием разрабатываемого инструмента от профессиональных аналогов является его педагогическая направленность. В реализуемом плагине нейросеть должна выступать в роли интеллектуального помощника и преподавателя. Данная логика заложена на уровне параметризации системных промптов.

Система содержит конфигурационный параметр, определяющий режим помощи (Help Mode), который может меняться в зависимости от текущих образовательных целей:

* 	Режим "Advice" (Наводящая подсказка): используется для ситуаций, когда студенту необходимо самостоятельно реализовать код на основе комментария. Такая конфигурация подходит для самопроверки или выполнения работы в режиме мониторинга, где допускаются подсказки. Системный промпт запрещает генерацию готового программного кода. Модель инструктируется анализировать ошибку и возвращать только краткий текстовый совет или вопрос на естественном языке, направляя студента к самостоятельному поиску правильного решения.
* 	Режим "Code" (Пример реализации): применяется в ситуациях, когда студенту необходимо продемонстрировать правильный паттерн программирования (например, оптимальный способ итерации по словарю). В этом режиме промпт позволяет ИИ сгенерировать исправленный фрагмент кода в сопровождении с кратким комментарием.

Динамическое переключение данных режимов формирует адаптивную образовательную среду, снижая когнитивную нагрузку обучающегося без ущерба для академической самостоятельности.
